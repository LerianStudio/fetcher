1 · Estado Atual — antes desta mudança

Hoje, o fetcher possui um endpoint mínimo de liveness no serviço manager e absolutamente nada no worker. Não existe probe de prontidão conectado ao Kubernetes, nenhuma agregação de dependências, nenhum self-probe, nenhuma validação de postura TLS e nenhum 503 acoplado ao drain. Um restart de qualquer dependência downstream pode deixar qualquer um dos serviços aceitando tráfego que não consegue atender.

pkg/bootstrap/readyz/handler.go (NOVO — esqueleto)
package readyz

import (
    "context"
    "net/http"
    "sync"
    "time"

    "github.com/gofiber/fiber/v2"
)

func (d *Dependencies) Handler() fiber.Handler {
    return func(c *fiber.Ctx) error {
        // 1. Short-circuit de drain — o K8s precisa parar de rotear para este pod AGORA.
        if IsDraining() {
            return c.Status(http.StatusServiceUnavailable).JSON(ReadyzResponse{
                Status: "draining",
                Version: d.Version, DeploymentMode: d.Mode,
            })
        }

        ctx, cancel := context.WithTimeout(c.Context(), 2*time.Second)
        defer cancel()

        // 2. Probes paralelos por dependência (cada um respeita seu próprio sub-timeout + carve-out MT).
        var wg sync.WaitGroup
        results := make(map[string]DependencyStatus, len(d.Probers))
        var mu sync.Mutex

        for name, probe := range d.Probers {
            wg.Add(1)
            go func(name string, p Prober) {
                defer wg.Done()
                start := time.Now()
                ds := p.Probe(ctx)
                d.Metrics.ObserveCheck(name, ds.Status, time.Since(start))
                mu.Lock(); results[name] = ds; mu.Unlock()
            }(name, probe)
        }
        wg.Wait()

        // 3. Agregação — qualquer "down" => 503; todos "up"/"skipped"/"n/a" => 200.
        overall, httpCode := aggregate(results)

        return c.Status(httpCode).JSON(ReadyzResponse{
            Status:         overall,
            Checks:         results,
            Version:        d.Version,
            DeploymentMode: d.Mode,
        })
    }
}
pkg/bootstrap/readyz/tls_enforcement.go (NOVO — esqueleto)
package readyz

import (
    "errors"
    "fmt"
)

var ErrSaaSPlaintextDep = errors.New("saas mode requires TLS on all platform dependencies")

func ValidateSaaSTLS(cfg *Config) error {
    if cfg.DeploymentMode != ModeSaaS {
        return nil  // byoc / local — a postura é apenas informativa
    }

    checks := []struct {
        name string
        tls  func(*Config) bool
        skip func(*Config) bool
    }{
        {"mongodb",       detectMongoTLS,        nil},
        {"rabbitmq",      detectAMQPTLS,         nil},
        {"redis",         detectRedisTLS,        redisSkipped},
        {"s3",            detectS3TLS,           s3Skipped},
        {"tenant_manager", detectHTTPUpstreamTLS, tenantMgrSkipped},
    }

    for _, ck := range checks {
        if ck.skip != nil && ck.skip(cfg) { continue }
        if !ck.tls(cfg) {
            return fmt.Errorf("%w: %s not using TLS", ErrSaaSPlaintextDep, ck.name)
        }
    }
    return nil
}

Regra de agregação

O status geral é healthy (HTTP 200) somente quando cada dependência está up, skipped ou n/a. Qualquer único down ou degraded força unhealthy e HTTP 503. Isso segue o padrão readyz do Ring e mantém o roteamento do K8s alinhado à capacidade real.

6 · Cerca de Escopo — considerado, e então excluído

Estes itens foram avaliados e deliberadamente deixados de fora. Cada um tem uma razão específica e nenhum pertence dentro do /readyz.

ConsideradoStatusPor que está fora de escopo
Probe de PostgreSQL EXCLUÍDO o fetcher usa Postgres apenas como datasource do usuário (DBs externos de clientes via conectores de ingestão de dados). Não é estado interno da plataforma. A saúde dele é problema do cliente, não nosso.
Verificações sintéticas de transação de negócio EXCLUÍDO Prontidão de infraestrutura ≠ validade de negócio. Transações sintéticas pertencem a canários / monitoramento sintético, não ao /readyz.
Validação de expiração de certificado EXCLUÍDO Reportamos a postura TLS (ligado/desligado) no momento do probe. Monitoramento de expiração é uma preocupação de cert-manager / alerta, fora do orçamento de latência do probe.
SLIs de latência p99 no /readyz EXCLUÍDO Percentis de latência vivem em dashboards do Grafana (o histogram readyz_check_duration_ms é suficiente para isso). O probe reporta uma única amostra de latência, não um agregado.
Regras de negócio por tenant no /readyz/tenant/:id EXCLUÍDO O endpoint de tenant é apenas de infraestrutura: o Mongo e o Rabbit do tenant funcionam? A validação de regras de negócio pertence ao interior da aplicação, não ao probe.
Substituir /readyz por commonsHttp.HealthWithDependencies EXCLUÍDO Esse helper cobre apenas a agregação de dependências. Ele não fornece: gating por self-probe, 503 acoplado ao drain, gauge Prometheus de self-probe, carve-out multi-tenant, nem postura TLS na resposta. Usá-lo deixaria seis lacunas.

7 · Avaliação de Riscos — cinco riscos canônicos, mitigados

RiscoSeveridadeMitigação (específica do fetcher)Verificado em
/readyz acidentalmente atrás do middleware de auth MÉDIA O auth no fetcher-manager é anexado por grupo de rotas, não globalmente. /readyz é montado na L53 no grupo público entre /health e /version — no mesmo nível dos endpoints sem auth existentes. routes.go:52–55 · teste e2e afirma que não há 401 sem token
Grace de drain curto demais — K8s ainda roteia no shutdown MÉDIA O default READYZ_DRAIN_DELAY_SEC=12s satisfaz periodSeconds=5 × failureThreshold=2 = 10s mais 2s de buffer. Ajustável por env. Teste de caos valida a propagação SIGTERM→503 dentro da janela de grace. teste de caos readyz_chaos_test.go · default do Helm terminationGracePeriodSeconds ≥ 30
Modo SaaS bootando com dep em plaintext (downgrade silencioso de TLS) ALTA ValidateSaaSTLS é chamado em manager config.go:196 e worker config.go:153 antes da primeira chamada a initMongoRepositoriesFn / initStorageRepository. Falha com ErrSaaSPlaintextDep envolvendo o nome da dep. teste unitário tls_enforcement_test.go por dep · e2e boota SaaS com Mongo em plaintext → espera fatal
Métricas Prometheus registradas mas nunca emitidas MÉDIA Cada chamada de probe roda d.Metrics.ObserveCheck(name, status, duration) dentro do handler. APROVAÇÃO DO GATE 1.5 NECESSÁRIA

As sete decisões abaixo estão pré-aprovadas. Confirme para destravar o Gate 2 (scaffolding).

  • Escopo — apenas deps internas de plataforma
    MongoDB · RabbitMQ · Redis · S3 (apenas worker) · Tenant Manager HTTP (apenas MT). Postgres excluído (datasource do usuário, não estado da plataforma).
  • HEALTH_PORT=4007 do worker
    Novo micro-servidor Fiber monta /health, /readyz, /metrics via um segundo commons.RunApp em service.Run, compartilhando o shutdown gracioso com o consumer.
  • Carve-out multi-tenant
    O /readyz global reporta Mongo + RabbitMQ como n/a com ponteiro para /readyz/tenant/:id. O endpoint de tenant valida via tmclient.GetActiveTenantsByService e resolve managers por tenant. Sem auth — o id do tenant na URL serve como prova de posse.
  • Tratamento de circuit breaker
    Tenant Manager: estado opaco, errors.Is(ErrCircuitBreakerOpen) mapeia para down. Adaptador RabbitMQ: CircuitBreakerState() → closed/half-open/open → up/degraded/down.
  • Enforcement de TLS em SaaS
    DEPLOYMENT_MODE=saas|byoc|local (default local). ValidateSaaSTLS() roda antes de qualquer conexão em manager:196 e worker:153. Dep em plaintext aborta o boot em SaaS.
  • Grace de drain = 12s
    O default READYZ_DRAIN_DELAY_SEC=12 satisfaz periodSeconds=5 × failureThreshold=2 + 2s do K8s. Ajustável por env em cada ambiente.
  • Prioridade da fonte de versão
    env OTEL_RESOURCE_SERVICE_VERSION → env VERSION → "unknown". Já parseado em cfg.OtelServiceVersion em manager config.go:68 / worker config.go:66.
